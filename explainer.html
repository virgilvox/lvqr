The State of Streaming is a Dumpster Fire

There are a million streaming servers out there. Ant Media Server (Java, 500MB+ memory baseline). Wowza (Java, enterprise pricing). Owncast (Go, shells out to ffmpeg for transcoding). MediaMTX (Go, lightweight but no fanout scaling). Mux (SaaS, pay-per-minute).

They all share the same fundamental problem. Data touches userspace too many times. Ingest arrives on a socket, gets copied to a buffer, processed, copied again, written to HLS segments on disk, read back, copied to response buffers, sent out. Every copy burns CPU. Every copy burns memory. Every copy is a ceiling on your concurrent viewer count.

The Architecture That Makes It Possible

LVQR is built on four pillars. Each one eliminates an entire category of overhead that every existing streaming server pays.

LVQR Stack (click layers to explore)
Click a layer to learn how it works.

io_uring

Batched async I/O with sendmsg_zc for payloads over 10KB. Eliminates syscall overhead by batching hundreds of network ops into a single submission queue. Combined with ref-counted buffers, the hot path stays lean.

QUIC / MoQ

Media over QUIC Transport (draft-17, April 2026). Pub/sub with named tracks. No head-of-line blocking. Independent stream multiplexing. Cacheable at relay nodes. Sub-500ms latency. Cloudflare has deployed it globally across 330+ cities.

Peer Mesh

Viewers relay to other viewers via WebRTC DataChannels. Quanteec proved 75-90% CDN offload in production at the Tour de France, Roland Garros, and Le Mans in 2025. LVQR bakes this into the protocol layer on top of MoQ.

Ring Buffers

mmap'd shared ring buffers hold GOPs. Ref-counted bytes::Bytes slices mean subscribers read from the same buffer, no copy. Thread-per-core shared-nothing design. Each core owns its subscriber set. No locks on the hot path.

The key insight is that a streaming relay doesn't need to understand the media. MoQ treats video as opaque named tracks. The relay just moves bytes from publishers to subscribers. That means the relay can be a minimal-copy data pump, sharing buffer references instead of duplicating data.

Minimal-Copy: The Data Path That Actually Works

Here's the traditional flow. An RTMP ingest arrives. The server reads it into a buffer. Parses it. Writes HLS segments to disk. When a viewer requests a segment, it reads from disk into a buffer. Writes that buffer to the response socket. That's four copies per viewer per segment.

LVQR's flow: QUIC ingest arrives. Quinn decrypts and deframes in userspace (QUIC is a userspace protocol, this is unavoidable). The payload lands in a ref-counted bytes::Bytes buffer inside an mmap'd ring. When subscribers need data, they get a cheap reference to the same buffer. io_uring's sendmsg_zc pushes it to the wire without an extra kernel copy. One copy on ingest, zero per additional subscriber.

Honesty check: QUIC encryption happens in userspace. You can't splice encrypted QUIC payloads kernel-to-kernel like you can with raw TCP sendfile(). The win here is ref-counted buffer sharing (one buffer, many readers) plus io_uring batched sends, not magical kernel bypass.

Viewers Become The CDN

Even with minimal-copy, a single server has a bandwidth ceiling. A $6 droplet has maybe 1Gbps outbound. At 4Mbps per viewer, that's 250 viewers max. Sounds like a hard wall.

Unless your viewers relay to each other. LVQR builds a peer mesh. The server seeds the stream to a handful of "root" viewers. Those viewers relay to more viewers via WebRTC DataChannels. The mesh self-organizes. The server's bandwidth multiplies exponentially.

This isn't theoretical. Quanteec runs this in production for France Télévisions. At the 2025 Tour de France, they hit 75% average CDN offload. At GP Explorer 3, 86% average, 88% peak. At Le Mans 2025, 79% offload with 0.03% rebuffering via peers vs 0.86% via CDN alone.

The Crate Map

LVQR builds on existing battle-tested MoQ libraries. The moq-lite transport layer (from moq-dev/moq, 1000+ stars) already works with Cloudflare's CDN. LVQR adds the peer mesh and optimized relay on top.

Crate Architecture

Deploy: scp lvqr server:/usr/local/bin/ and you're done.

Usage: lvqr serve --port 4443 --mesh-enabled --max-peers 3. OBS points RTMP at port 1935. Viewers connect via WebTransport (all modern browsers, including Safari 26.5+). Everyone gets sub-second latency.

What Makes This Actually Novel

MoQ exists. io_uring exists. Peer mesh exists. The trick is nobody has combined MoQ relay + peer mesh in a single system. Cloudflare's relay is centralized CDN. Quanteec's mesh runs on HLS/DASH. Each piece amplifies the others.

The Multiplier Effect

io_uring + MoQ: MoQ's opaque track model means the relay never parses media. io_uring's batched submission and sendmsg_zc minimize syscall overhead. Other MoQ implementations use standard tokio. LVQR uses io_uring on the hot send path.

Peer mesh + MoQ: MoQ's named-track pub/sub maps perfectly onto a peer relay tree. Peers subscribe to tracks by name. The mesh is just another layer of MoQ relay nodes running in browsers via WebTransport.

io_uring + peer mesh: The server only serves ~30 root peers instead of 5000 viewers. io_uring never gets saturated. Dozens of simultaneous channels on one core.

OBS, Streamlabs, and every mobile encoder speak RTMP. LVQR accepts RTMP and translates to MoQ tracks internally. An OBS plugin for native MoQ output already exists (moq-dev/obs).

For developers: REST API for stream management, auth tokens, webhook events. The web player is ~30KB gzipped, handles transport negotiation automatically. One script tag.

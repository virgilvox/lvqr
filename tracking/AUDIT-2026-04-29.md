### Wire-tested PASS

- **C-3 dynamic MPD** -- `curl http://127.0.0.1:8889/dash/live/<bcast>/manifest.mpd`
  on a live ffmpeg RTMP publisher renders the spec-mandated trio:
  `availabilityStartTime="2026-05-01T05:03:27.597Z"`,
  `publishTime="2026-05-01T05:03:32.525Z"`, and a
  `<UTCTiming schemeIdUri="urn:mpeg:dash:utc:direct:2014" value="..."/>`
  descriptor placed AFTER the Period element per ISO/IEC 23009-1
  §5.3.1.2 child-element ordering. `type="dynamic"`,
  `profiles="urn:mpeg:dash:profile:isoff-live:2011"`,
  `minimumUpdatePeriod="PT2.0S"` all present. Post-finalize the same
  MPD correctly switches to `type="static"` with the on-demand

  are wire-suppressed when unset, which is the C-3 contract).

### Operator finding (informational, not a regression)

`/api/v1/server-info` on a relay booted without any auth flags
returns `auth_mode: "configured"`, even though the boot log says
`auth: open access (no tokens configured)`. The session-167 fix
(`0a8d740`) ladders to `"noop" | "static" | "jwt" | "jwks" |
"webhook" | "configured"`, but the default `--no-streamkeys` path
wraps the configured provider in `MultiKeyAuthProvider` for the
streamkeys CRUD, which appears to flip the classifier off the
"configured" boot bucket regardless of the underlying provider
being Noop. Worth a follow-up classifier pass that detects the
wrap-only-for-streamkeys case and reports the inner provider's
mode.

### Lvqr-dash 1.0.0 SemVer ergonomics fix landed

The session-168 deferral on the C-3 struct-literal break is
closed. `Default` impls land on `Mpd` / `Period` / `AdaptationSet`
/ `Representation` / `SegmentTemplate` / `MpdType`. External
embedders who built against pre-C-3 1.0.0 can upgrade to the
C-3-bearing release with `Mpd { periods, ..Default::default() }`
and stay forwards-compatible against future optional-field
additions. Defaults match LVQR's in-tree live-profile values
(`MpdType::Dynamic`, `urn:mpeg:dash:profile:isoff-live:2011`,
`PT2.0S` / `PT2.0S` for `min_buffer_time` / `minimum_update_period`,
all four post-C-3 timing fields default to `None` so the wire
shape matches the pre-C-3 shape byte-for-byte). New unit test
`default_spread_lets_embedder_supply_only_meaningful_fields`
locks the spread pattern in. 38 lvqr-dash lib tests pass; fmt
clean; clippy clean.

## Status update -- 2026-04-30 session 169 close (C-9)

One more critical closed.

- **C-9 WHEP 422 upfront when publisher audio is unservable** --
  `b39901e` `fix(whep)`. New default trait method
  `SdpAnswerer::supports_audio_codec(codec)` returns `true` only
  for Opus by default; `Str0mAnswerer` overrides to also accept
  AAC when the `aac_opus_factory` is wired (cfg-gated on
  `aac-opus`). Router's `handle_offer` reads the broadcast's

  short-circuits to 422 when the publisher's codec is not in the

  §15.5.21 422 Unprocessable Entity (offer is well-formed; the
  resource cannot serve it). Best-effort gate: when no audio
  config has been published yet, the cache is empty and the gate
  falls through; the runtime per-sample
  `lvqr_whep_codec_mismatch_drops_total` counter (audit I-7)
  remains the canonical drop-rate source. New Prometheus
  counter `lvqr_whep_audio_codec_unavailable_total{broadcast,
  codec}`. Three new integration tests: AAC publisher -> 422,
  Opus publisher -> 201, no-config-yet -> 201. 17 lvqr-whep
  integration tests pass; 27 lib tests pass; clippy + fmt clean.

## Critical

### C-1. WHIP/WHEP `Location` header unreadable from browser JS (CORS)

**Spec violation**: WHIP draft-ietf-wish-whip-15 §4.1 (and the same in
WHEP). The 201 Created response carries `Location:` pointing at the
session resource URL; the client uses that URL for PATCH (trickle ICE)
and DELETE (terminate). The browser Fetch / XHR layers refuse to expose
non-CORS-safelist response headers to JS unless the server lists them in
`Access-Control-Expose-Headers`. `Location` is **not** on the safelist.

**Evidence**: `crates/lvqr-cli/src/lib.rs` lines 1457 + 1477 mount
`CorsLayer::permissive()` on the WHIP and WHEP routers.
`tower_http::cors::CorsLayer::permissive()` permits the request side
(`allow_origin: any`, `allow_headers: any`, `allow_methods: any`) but
sets the response side `expose_headers` to **empty**. A browser-side
WHIP client can POST the offer and receive 201 + Location, but its JS
cannot read the header value, so PATCH and DELETE land on the wrong
URL or never fire.

**Why workspace tests do not catch it**: every WHIP / WHEP integration
test in the tree drives the protocol from a Rust client (`crates/lvqr-cli/tests/whip_hls_e2e.rs`,
`rtmp_whep_audio_e2e.rs`) where CORS does not gate header visibility.
The browser admin-ui's `useWhipPublisher.ts` happens to work today
because Chrome's WHIP draft implementation is permissive, but this is
incidental.

**Fix**: small (under 30 lines). Replace `CorsLayer::permissive()` on
the WHIP / WHEP / signal / HLS / DASH routers with an explicit layer:

```rust
use tower_http::cors::{Any, CorsLayer};
use http::header;
let cors = CorsLayer::new()
    .allow_origin(Any)
    .allow_methods(Any)
    .allow_headers(Any)
    .expose_headers([header::LOCATION, header::CONTENT_TYPE, header::ETAG]);
```

`ETAG` is added because the WHIP draft suggests `If-Match` for SDP
renegotiation, which requires a readable `ETag` on the 201 response.

### C-2. WHEP egress has no PLI / FIR keyframe-request feedback

**Spec violation**: WHEP draft-ietf-wish-whep-13 §4.2 mandates
RTCP feedback for keyframe requests. Without PLI / FIR, a viewer that
joins mid-stream waits for the next IDR (potentially up to GOP length)
before seeing any video, and bandwidth-drop recovery (decoder asks for
a refresh keyframe) never reaches the publisher.

**Evidence**: `crates/lvqr-whep/src/str0m_backend.rs` event loop (around
lines 689-706) handles `IceConnectionStateChange`, `Connected`,
`MediaAdded`, but has **no handler for `Event::KeyframeRequestRecv`**
or anything that proxies a PLI back through the
`FragmentBroadcasterRegistry` to the publisher. The SDP answer
advertises `a=rtcp-fb:* nack pli` but the inbound feedback events are
discarded.

**Fix**: large (cross-crate). Need a path from `str0m::Event::KeyframeRequest`
on the WHEP session to an upstream `KeyframeRequest` signal on the
publisher's ingest path. RTMP: there is no native PLI in RTMP, but we
can rotate the encoder via the FLV `onFI` mechanism; in practice
`FragmentBroadcaster` owns the encoder and can emit a "next sample
must be IDR" hint that the WHIP / RTMP / SRT ingest paths honor.

### C-3. DASH MPD missing `availabilityStartTime`, no `UTCTiming` element

**Spec violation**: ISO/IEC 23009-1:2022 §5.3.1.2: for live
(`type="dynamic"`) MPDs, `availabilityStartTime` is required, and a
`UTCTiming` descriptor (or the `availabilityStartTime` plus a
synchronized clock) is required for clock alignment.

**Evidence**: `crates/lvqr-dash/src/mpd.rs:417-468` emits the `<MPD>`
element with `type` and `mediaPresentationDuration` but no
`availabilityStartTime` attribute and no `<UTCTiming>` child.

**Player impact**: dash.js, Shaka, ExoPlayer all refuse to attach a
live presentation timeline to a dynamic MPD without this. The MPD
either parses as VOD-styled (no live edge) or the player drops out
with a "missing AST" error.

**Fix**: large. Need to thread the wall-clock anchor (the moment the
encoder began producing the first sample of the presentation) through
the bridge and emit `availabilityStartTime` as
`format!("{}", chrono::Utc::now().to_rfc3339_opts(SecondsFormat::Millis, true))`.
For low-latency DASH, also emit `availabilityTimeOffset` per
representation.

### C-4. HLS `EXT-X-TARGETDURATION` not derived from real segment durations

**Spec violation**: RFC 8216bis §4.4.3.1: TARGETDURATION must be the
**ceiling** of the longest segment in the playlist, in integer seconds.

**Evidence**: `crates/lvqr-hls/src/manifest.rs:258` writes
`#EXT-X-TARGETDURATION:{}` from `self.target_duration_secs` (a `u32`
configured at builder construction). The builder never inspects the
durations of pushed segments to recompute. If a segment is 2.001 s but
config says 2 s, strict players reject the manifest with
`MediaError: HLS_MANIFEST_PARSING_ERROR`.

**Player impact**: hls.js logs a fatal `MANIFEST_PARSING_ERROR` and
stops; iOS Safari (the strictest production HLS client) fails the
playback altogether.

**Fix**: medium (single file). Track the max-observed `partial_target *
parts_per_segment` rounded to integer seconds, ceil it, and emit that
instead of the configured value. Backwards-compatible: the config
value becomes a floor.

### C-5. HEVC codec string emits `general_profile_compatibility_flags` in the wrong bit order

**Spec violation**: RFC 7798 §3.2 (HEVC RTP payload format,
`HEVCDecoderConfigurationRecord` codec-string mapping) and ISO/IEC
14496-15 §8.3 specify the compat-flags field is serialised in
**reverse bit order** (LSB-first) into the codec string. `format!(":X",
...)` outputs the field MSB-first.

**Evidence**: `crates/lvqr-codec/src/hevc.rs:120-134`:

```rust
pub fn codec_string(&self) -> String {
    format!(
        "hev1.{}{}.{:X}.L{}.B0",
        match self.general_profile_space { ... },
        self.general_profile_idc,
        self.general_profile_compatibility_flags,
        self.general_level_idc,
    )
}
```

**Player impact**: HEVC publishers feeding Shaka / dash.js are
rejected with codec-not-supported. iOS Safari is more forgiving (uses
`hvc1` sample-entry only) so this can hide depending on which player
the operator tests against.

**Fix**: small (under 30 lines). Bit-reverse the 32-bit field before
the `{:X}` format. Add a unit test against a known SPS (e.g. the one
in `crates/lvqr-codec/tests/fixtures/`).

### C-6. WHIP / WHEP accept SDP offer with unsupported codec, then silently drop

**Spec violation / interop risk**: WHIP draft §3.1: when the server
cannot answer an offer compatible with its capabilities, it should
return 4xx (typically 415 Unsupported Media Type) or answer with
`a=inactive` on the offending m-line.

**Evidence**: `crates/lvqr-whip/src/str0m_backend.rs` accept_offer
(lines 81-146) builds an answer from str0m, which silently negotiates
out codecs the bridge does not handle. The session POST returns 201,
the publisher sees ICE connect, but no MediaData ever flows because
the bridge's `MediaCodec` match (line 352-359) drops anything other
than H.264 / HEVC. The recently-added browser-side `setCodecPreferences`
fix (commit d14b726) papered over this but does not address the
root cause.

`crates/lvqr-whep/src/str0m_backend.rs:798-818` has the same shape:
audio AAC -> Opus-only subscriber drops, video H.265 -> H.264-only
subscriber drops, with a `tracing::warn!` that fires once per session
then silences.

**Fix**: medium. Add a pre-answer codec validator that intersects the
offer's codecs with the bridge's accepted set; if empty, return HTTP
415. Same on WHEP (the answerer should refuse if no compatible codec
is present in the publisher).

### C-7. HLS `EXT-X-DISCONTINUITY` not emitted when publisher reconnects with new init

**Spec violation**: RFC 8216bis §4.4.4.4: a discontinuity tag MUST
appear when the codec or encoding parameters of consecutive segments
differ.

**Evidence**: `crates/lvqr-cli/src/hls.rs:146-167` detects an init
segment change and calls `server.push_init(...)` + resets
`CmafPolicyState`; **but** `crates/lvqr-hls/src/manifest.rs:611-632`
keeps incrementing `next_sequence` monotonically and never emits
`#EXT-X-DISCONTINUITY`. Strict clients (hls.js, Shaka) glitch or
fail playback at the boundary.

**Fix**: small (under 30 lines). Add a `discontinuity: bool` field on
the next-segment-to-emit row inside `PlaylistBuilder` and have the
HLS bridge set it when `current_init != last_init`. Render path: emit
`#EXT-X-DISCONTINUITY` immediately before the segment URI line.

### C-8. Live HLS / DASH `CorsLayer` posture inconsistent

**Spec context**: Apple HLS authoring spec + DASH-IF guideline both
require `Access-Control-Allow-Origin: *` (or equivalent) on every
manifest + segment URL when the player is browser-side.

**Evidence**: `crates/lvqr-hls/src/server.rs:7-9` documents:

```
//! no CORS (the deployment adds that at the reverse
//! proxy), no auth (auth lives in `lvqr-auth` and slots in via the
//! `lvqr-cli` serve path when HLS joins the composition root).
```

The CLI composition root mounts `CorsLayer::permissive()` on the
combined HLS + DASH router (line 1457). Permissive is fine for the
mainline browser-fetch case but **does not expose `Access-Control-Allow-Headers`
on the actual HLS / DASH 200 responses for non-OPTIONS verbs**, which
in practice is fine because GET / HEAD never need custom headers.

**Conclusion**: works for browser playback today. The risk is that the
`CorsLayer::permissive()` choice silently drops a future custom
header (e.g. `X-Lvqr-Latency-Hint`) needed by client SDKs. Annotate
the routers to use the explicit form from C-1 above.

### C-9. WHEP AAC -> Opus mismatch silently drops audio when `transcode` feature is off

**Evidence**: `crates/lvqr-whep/src/str0m_backend.rs:798-804` warns
once per session then drops every subsequent AAC sample. RTMP-publisher
to WHEP-subscriber on a build without `--features transcode` produces
a video-only stream; the operator sees `ON AIR` but no audio.

**Fix**: medium. The `--features transcode` builds already ship the
session 113 AAC->Opus path. For non-transcode builds: emit an explicit
HTTP 422 on the WHEP POST when the publisher is AAC-only and the
session would have to drop audio.

## Important

### I-1. WHIP trickle ICE PATCH is logged but not applied

**Evidence**: `crates/lvqr-whip/src/str0m_backend.rs:465-470` --
PATCH body is parsed, candidate fragments are extracted, but the
candidate is **logged at warn level and discarded**. The session
proceeds with whatever ICE candidates the original offer carried.

**Impact**: a publisher behind a NAT that needs to send mid-session
candidates (network change, Wi-Fi to LTE) cannot do so. WHIP draft
§4.2 says trickle SHOULD be supported.

**Fix**: medium. str0m does support post-init `add_remote_candidate`;
wire the PATCH path to it.

### I-2. WHEP router calls `extract_whip()` for auth

**Evidence**: `crates/lvqr-whep/src/router.rs:99` calls the function
named `extract_whip`. Functional today (both protocols extract the
Bearer header the same way) but a future change to WHIP-side auth
that does not apply to WHEP would silently regress WHEP.

**Fix**: small. Rename `extract_whip` to `extract_session_token` (or
similar protocol-agnostic name) and re-export under both legacy
names for one release cycle.

### I-3. SRT unknown TS stream type silently dropped at debug level

**Evidence**: `crates/lvqr-srt/src/ingest.rs:248-250` logs
`debug!("SRT unknown stream type; dropping PES")`. Operators running
without `RUST_LOG=debug` see no warning when SRT carriage of HEVC
(stream_type 0x24) or AV1 (0x42) is dropped.

**Fix**: small. Elevate to `warn!` and rate-limit (one per stream_type
per minute per broadcast).

### I-4. RTSP unknown rtpmap codec silently falls back to H.264 drain

**Evidence**: `crates/lvqr-rtsp/src/server.rs:954-990` dispatches
between `play_drain_h264` and `play_drain_hevc` based on init segment
shape; for non-H.264 / non-HEVC, **falls back to H.264 unconditionally**.
Result: the depacketizer fails silently; PLAY appears to start but
emits nothing.

**Fix**: small. Return RTSP 415 in DESCRIBE when the broadcast init
is neither AVC nor HEVC.

### I-5. RTMP video_codec_id not validated against payload format

**Evidence**: `crates/lvqr-ingest/src/rtmp.rs:389` logs the metadata
but never rejects non-7 (H.264) values. Enhanced RTMP HEVC
(`fourCC: hvc1`) and AV1 (`fourCC: av01`) work today only because
the RTMP bridge's bytestream parsing is shape-agnostic; if a
publisher claims H.264 in metadata but pushes VP6, the bridge
forwards garbage to the fragment registry.

**Fix**: small. Validate codec_id against the payload's first byte
and reject mismatches via `onStatus(error)` RTMP message.

### I-6. WHEP / WHIP rtcp-fb advertised but unused

**Evidence**: WHEP SDP answer carries `a=rtcp-fb:* transport-cc nack pli`
but the str0m event loop ignores RTCP feedback events.
Bandwidth-aware ABR cannot land without consuming these.

### I-7. WHEP non-H.264/HEVC video silently logged-once at warn

**Evidence**: `crates/lvqr-whep/src/str0m_backend.rs:814-818`. After
the one-shot warn, every subsequent sample is discarded silently.
This is the same shape as the original VP8 silent-drop bug; the warn
is a soft compensation but the operator metric story is missing.

**Fix**: small. Increment a `lvqr_whep_codec_mismatch_drops_total`
counter on every drop, labeled by `(broadcast, codec)`. Operator
sees the drop rate on Prometheus.

### I-8. Federation runner observes only remote MoQ announcements

**Evidence**: `crates/lvqr-cluster/src/federation.rs:573-594` drains
the remote cluster's MoQ announcement stream. A federation-only
inbound broadcast does not appear in the local
`/api/v1/streams` response unless re-published locally. By design,
but undocumented.

**Fix**: small. Add a per-link "imported broadcasts" tally to the
federation status surface.

## Competitor-feature gap analysis

Surveying OvenMediaEngine, Mediasoup, LiveKit, SRS for protocol-level
features we lack and that operators routinely ask for.

### WHIP/WHEP

1. **OvenMediaEngine** ships full simulcast pass-through on WHIP and
   layer selection on WHEP. We have none of it; every WHEP subscriber
   gets the single quality the publisher pushes.
2. **LiveKit** ships server-side bandwidth estimation on WHEP and
   adaptive simulcast layer pinning. We do not consume any RTCP
   feedback (I-6).
3. **Mediasoup** treats codec capability as a first-class
   `RtpCapabilities` object that the client negotiates against
   before SDP is built. Our bridge does the negotiation inside
   str0m, which is why codec mismatches fail silently (C-6).

### HLS

1. **OvenMediaEngine** + **Mux** both ship LL-HLS with sub-300ms
   glass-to-glass on real ingests. Our LL-HLS scaffolding is in
   place (PART, PRELOAD-HINT, RENDITION-REPORT, CAN-BLOCK-RELOAD)
   but C-4 + C-7 mean strict players reject our output before
   measuring latency.
2. **SRS** ships HLS with `EXT-X-DATERANGE` for ad markers (SCTE-35
   passthrough). Anti-scoped per ROADMAP but worth re-evaluating.
3. **Apple HTTP Live Streaming Tools** ship a `mediastreamvalidator`
   binary; running it against our output in CI would catch C-4 +
   C-7 + most B-5 class issues before merge.

### DASH

1. **OvenMediaEngine** ships full live LL-DASH with
   `availabilityTimeOffset` per Representation; we lack
   `availabilityStartTime` entirely (C-3).
2. **dash-IF dashif-conformance** Docker container is the
   authoritative MPD validator; PLAN_V1.1 rows 117/118 noted this
   as deferred. Time to wire it in.
3. **Shaka Streamer** ships chunked encoding with consistent
   `presentationTimeOffset` across rotations. Our DASH does not
   handle encoder rotation cleanly (no DISCONTINUITY equivalent).

## Scoping decision locked in-briefing -- keep the MoQ wire pure

The session 109 A briefing listed three design options for WS /
MoQ / WHEP propagation: (a) carry `ingest_time_ms` on a MoQ frame
header, (b) have the drain subscribe additionally to the fragment
registry purely for SLO sampling, or (c) a sidecar `SloSampler`
with its own wall-clock heuristics. **Decision for 110: option
(b).** Rationale:

1. **`moq_lite` 0.15 frames are bare `Bytes` with no header /
   extension capability**. The only wire-level propagation option
   is payload framing -- prepend an 8-byte u64 to every
   `group.write_frame(bytes)` call in
   `crates/lvqr-fragment/src/moq_sink.rs::MoqTrackSink::push`.
   Doing so would bloat every frame by 8 bytes (a low
   information-density tax for a server-internal concern) AND
   break any foreign MoQ client that decodes frame payloads as
   raw CMAF bytes. LVQR's own browser playback stack would need
   an update simultaneously; a third-party MoQ player would break
   silently. The SLO tracker is a server-internal observability
   surface. It should not leak into a public wire protocol.

2. **Federation's forward path copies frames verbatim**
   (`crates/lvqr-cluster/src/federation.rs` `forward_track()`),
   so a payload-prefix approach would be transparent to
   federation specifically. That's fine but not a reason to take
   the prefix route -- it's the only redeeming property.

3. **The `FragmentBroadcasterRegistry` is the already-established
   server-internal fanout spine**. HLS, DASH, archive, WASM
   filter tap, cluster claim renewer, agent runner, and the
   captions bridge all subscribe on it via
   `on_entry_created`. Adding one more auxiliary subscription on
   the WS relay side for SLO sampling fits the existing
   pattern exactly, with no new abstractions.

4. **WHEP is actually a simpler case** that does not touch MoQ at
   all: WHEP's `SessionMsg::Video { payload, dts, keyframe, codec
   }` channel in `crates/lvqr-whep/src/str0m_backend.rs` is fed
   by the ingest bridge directly, bypassing MoQ fanout. Add
   `ingest_time_ms: u64` to the `SessionMsg::Video` variant and
   the WHEP drain records samples at the `rtc.writer(mid).write`
   call site with the same `saturating_sub` shape the HLS and
   DASH drains use. No MoQ wire involvement.

5. **Pure MoQ subscribers are genuinely out of server-side
   measurement scope.** They consume `OriginProducer` directly
   with no server-side drain task to hook. Their latency is
   measured in the client SDK (Tier 5 scope), exactly per the
   "server-side measurement only" bullet already documented in
   `docs/slo.md`.

## Session 110 scope -- two deliverables

### 110 A -- WS relay SLO instrumentation

**What changes:**

1. `crates/lvqr-cli/src/lib.rs::ws_relay_session` gains a
   fragment-registry argument (thread `shared_registry` through
   from `start()`; it already exists in scope) plus the shared
   `LatencyTracker`.
2. Inside `ws_relay_session`, for each `(broadcast, track)`
   pair the session resolves, additionally call
   `shared_registry.get_or_create(broadcast, track, meta)` (the
   broadcaster is guaranteed to exist because the ingest side
   already created it) and subscribe via `bc.subscribe()`. Spawn
   one auxiliary task per track that drains `BroadcasterStream`
   purely for `Fragment::ingest_time_ms` values; correlate with
   the MoQ frame drain via a per-track mpsc or `tokio::sync::watch`
   channel.

   **Simpler alternative to consider during 110 A planning**:
   instead of two subscriptions, replace the MoQ-side drain
   (`relay_track` reading `TrackConsumer`) with a fragment-registry
   drain (`BroadcasterStream::next_fragment`) that sends the
   fragment's `payload` verbatim onto the WS. This sidesteps the
   correlation problem entirely because each fragment carries
   both `payload` (for WS wire) and `ingest_time_ms` (for SLO
   record). The downside is that the WS relay no longer tests the
   MoQ fanout path; today's `relay_track` proves the MoQ
   subscription contract. Pick whichever keeps the test story
   cleanest -- in-commit design decision per the plan-vs-code
   rule.

3. In `relay_track` (or the fragment-registry-backed replacement),
   record one sample per outbound WS frame:

   ```rust
   if let Some(tracker) = slo.as_ref() && ingest_time_ms > 0 {
       tracker.record(&broadcast, "ws", now_unix_ms().saturating_sub(ingest_time_ms));
   }
   ```

   Skip zero `ingest_time_ms` (federation replays, synthetic test
   fragments). Transport label `"ws"` matches the threshold
   decision table row in `docs/slo.md`.

4. `WsRelayState` grows an `Option<LatencyTracker>` field (or
   receives the tracker by reference through the route's state
   extractor). The CLI composition root at the existing WS route
   registration line threads `slo_tracker.clone()` in.

## Key design decisions to bake in during session 110 (plan-vs-code rule)

(a) **Transport labels: `"ws"` + `"whep"`**. Unquoted lowercase,
    matches the existing `"hls"` + `"dash"` convention and the
    threshold decision table header in `docs/slo.md`.

(b) **No MoQ wire change.** `MoqTrackSink::push` must not grow
    an ingest-time prefix. Federation's `forward_track` stays
    unchanged. Any foreign MoQ client reading LVQR's output
    sees byte-identical frame payloads before and after 110.

(c) **Zero `ingest_time_ms` is still "unset".** All new drain
    points use the same `if ingest_time_ms > 0` guard the HLS
    and DASH drains already use.

(d) **Pure MoQ subscribers stay out of scope.** Document the
    gap in `docs/slo.md` "v1 limitations" as Tier 5 client SDK
    scope. Do not invent a sidecar sampler.

(e) **WS strategy: prefer fragment-registry drain over MoQ-side
    drain**, because the former gives `ingest_time_ms` for free
    without a second subscription. The trade is that the WS
    relay no longer exercises `TrackConsumer::next_group` as an
    integration path. If that integration coverage matters,
    keep the MoQ-side drain and add a second fragment-registry
    subscription for the stamp -- pick whichever keeps the test
    story cleaner, but bake the decision in-commit.

(f) **`SessionMsg::Video` field growth is additive.** Adding
    `ingest_time_ms: u64` to the struct variant is a
    source-compatible change inside `lvqr-whep`; external
    downstream crates that construct this variant directly
    (none today -- it's a private enum) would break. Audit the
    match arms to ensure every producer is updated.

## First decisions to lock before code

(i) **Does the WS relay switch to a fragment-registry drain, or
    keep the MoQ drain and add a second subscription?** This is
    the biggest branch. The fragment-registry drain is simpler
    and faster but loses MoQ-side integration coverage. Default
    recommendation: stay with the MoQ-side drain, add the
    fragment-registry subscription purely for the stamp, tie
    the two together via a per-`(broadcast, track)` inner
    channel. Reason: the WS relay today is the only in-tree
    integration test for the LVQR-side MoQ consumer API surface.

(ii) **WHEP tracker plumbing: constructor arg or context
     field?** If `Str0mBackend::new` takes an
     `Option<LatencyTracker>` it composes cleanly with the
     existing config-struct pattern. If the tracker lives on
     a `SessionContext` threaded through the poll loop, the
     WHEP test harness has less wiring churn. Default:
     constructor arg (matches HLS / DASH shape).

(iii) **WHEP integration test scope.** If a full WebRTC
      round-trip is prohibitively heavy for CI, accept a unit
      test that feeds the poll loop directly with a synthetic
      `SessionMsg::Video` and asserts the tracker snapshot.
      Document the compromise in the session 110 close block.

## Expected scope + biggest risks

~150-250 LOC across `crates/lvqr-cli/src/lib.rs` (WS relay
wiring + one new small helper), `crates/lvqr-whep/src/str0m_backend.rs`
(enum field growth + record call), both crates' tracker plumbing,
and the two new integration tests. Plus smaller edits on
`docs/slo.md` + `README.md`.

Risks, ranked:

1. **WS fragment-registry double-drain correlation**: if the
   WS relay keeps its MoQ-side drain AND adds a
   fragment-registry subscription for the stamp, the two can
   drift on group boundaries. Mitigation: a small per-track
   `VecDeque<u64>` of pending `ingest_time_ms` values drained
   alongside each `group.read_frame` call. Desync beyond a few
   frames is a symptom worth warn-logging.
2. **WHEP `SessionMsg::Video` producer audit**: missing a
   producer silently passes `Default::default()` or gets
   rejected by the compiler. Compiler catches it; the audit is
   just about how many call sites you touch.
3. **WHEP integration test heaviness**: if CI budget is tight,
   fall back to the inline unit test. Document the compromise
   in-close per the plan-vs-code rule.
4. **Pure MoQ subscribers complaining in code review**: someone
   may argue for the sidecar sampler or MoQ wire change. Point
   them at this briefing's scoping decision and the session 109
   briefing's three-option list. If the user wants to revisit
   after 110, a session 111 could add a sidecar sampler or a
   client-SDK collector.

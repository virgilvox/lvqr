# LVQR Competitive Audit -- 2026-04-13

A brutally honest comparison of LVQR as it stands today against the open and
closed-source media server field. Written after Tier 0 closed and before
Tier 1 work begins, with the goal of calibrating the roadmap against reality.

## Executive Summary

LVQR today is a focused Rust media server with a working
RTMP -> fMP4 -> MoQ -> WebSocket pipeline, a peer-mesh coordinator, a
pluggable auth layer, an admin HTTP/metrics surface, and one honest RTMP-to-WS
end-to-end test. Measured on feature surface it is roughly 10-15 percent of
MediaMTX, 5 percent of LiveKit, and a rounding error against Ant Media
Enterprise. Measured on architectural cleanliness and strategic positioning
it is in a genuinely interesting spot: MoQ-native, memory-safe, with a
credible plan to unify live and DVR under one fragment model.

The gap between "interesting prototype" and "server anyone deploys" is the
entire scope of Tiers 1 through 5 on the roadmap. None of it is impossible.
Most of it is execution. The only novel parts are the load-bearing
architectural decisions (Tier 2.1 through 2.3) that nobody else in the field
has attempted because MoQ itself is still stabilizing.

This document is not a victory lap. It exists so future sessions understand
what we are up against and do not waste cycles reinventing features the
competitive set solved five years ago.

## The Competitive Set

**Tier A -- the servers LVQR must eventually beat**

1. **MediaMTX** (Go, formerly rtsp-simple-server). The reference single-binary
   multi-protocol media server. Extremely popular for self-hosters. RTSP,
   RTMP, HLS, LL-HLS, WebRTC (WHIP/WHEP), SRT. Path-based model. On-demand
   pull of remote sources. Simple YAML config. Recording. Limited
   clustering. Active maintainer. MIT license.

2. **LiveKit** (Go, Pion WebRTC). The reference WebRTC SFU. Room model with
   participants and tracks. Simulcast, SVC, DTX, RED. Separate ingress
   service (RTMP, WHIP, URL pull) and egress service (HLS, MP4, RTMP, S3).
   Client SDKs for every platform. Managed cloud product with distributed
   routing. Agents framework for in-room AI. Apache 2.0 OSS plus enterprise
   tier. This is the most comprehensive OSS real-time video stack.

3. **AWS Kinesis Video Streams** (managed, closed). Storage-first CMAF
   fragments in a durable index. Producer SDK for embedded devices.
   GetMedia API streams fragments. HLS and DASH playback endpoints.
   Separate WebRTC signaling service. DVR window with long-term archive.
   Pay per GB ingested, stored, played. Strong in device/edge markets.

4. **Ant Media Server** (Java, Red5 fork origin). RTMP, WebRTC SFU, HLS,
   DASH, RTSP, SRT, MPEG-TS. Adaptive bitrate transcoding via ffmpeg.
   Conference rooms. Cluster with origin-edge. Free Community Edition,
   paid Enterprise with CMAF LL-HLS, DVR, WebRTC simulcast, CDN
   integration. Web admin UI. Big in mid-market.

**Tier B -- the servers LVQR overlaps with**

5. **OvenMediaEngine** (C++). Low-latency live broadcast focus. LL-HLS and
   WebRTC (WHIP/WHEP). RTMP, SRT, MPEG-TS ingest. In-process transcoder
   via ffmpeg libraries. Origin-edge clustering via their own OVT
   protocol. Signed policy tokens. Used widely in broadcast workflows.

6. **SRS** (C++, State Threads). Very popular in Asia. RTMP-first, with
   HLS/FLV/WebRTC/SRT as derived outputs. GB28181 support for Chinese
   video surveillance. High throughput per node. Edge clustering. DVR
   to FLV or MP4 files. External ffmpeg for transcoding.

7. **Nimble Streamer** (C++, closed). Free-tier multi-protocol relay.
   Strong in OTT distribution. SSAI. Commercial Softvelum variants.

8. **MistServer** (C++, commercial). Broadcast workflows. Wide protocol
   support. Commercial tier for advanced features.

9. **Janus Gateway** (C). WebRTC gateway with a plugin architecture.
   videoroom plugin is the most commonly deployed OSS SFU outside of
   LiveKit. Very stable, very mature, plugin SDK in Lua and Duktape.

10. **Jitsi Videobridge** (Java). The SFU underneath Jitsi Meet. Focused
    on conferences. Octo for geographically distributed rooms.

**Tier C -- adjacent/reference projects**

11. **moq-relay** (Rust, Luke Curley / kixelated). The MoQ reference
    relay. Subset of MoQ (moq-lite variant) that LVQR's `lvqr-relay`
    currently wraps via the moq-native crate. This is not a competitor;
    it is upstream.

12. **Cloudflare Media** (closed). Their internal live stack includes a
    MoQ-compatible relay at the edge, terminating moq-lite clients. They
    and Twitch are the current production MoQ consumers.

## Architecture Comparison

### MediaMTX

Single Go binary. Each protocol server lives in its own package. The central
abstraction is a Path -- every stream is named by a path like "live/foo".
Paths have sources (pushed by publishers or pulled on-demand from an
upstream URL) and destinations (readers connected via HLS, RTMP, WebRTC).
The Path Manager coordinates lifecycle. Format conversion is on-the-fly and
not persistent. Uses `gortsplib` for RTSP, a hand-rolled RTMP implementation,
Pion for WebRTC, and a hand-rolled fMP4 writer for HLS.

Key MediaMTX design decisions LVQR should study:

* **The Path model is the API.** One object owns the state of a stream from
  ingest to egress. MediaMTX exposes a JSON API for listing, adding, and
  removing paths at runtime. This is the ergonomic bar LVQR must clear.
* **On-demand pull.** `sourceOnDemand: yes` on a path causes MediaMTX to
  spawn a ffmpeg subprocess or open an RTSP client when the first viewer
  subscribes. LVQR has no equivalent concept.
* **The YAML config is the source of truth.** Hot reload via SIGHUP.
  Entire operator ergonomics flow from config file reloadability.
* **Reader/publisher asymmetry is minimal.** A single Path serves every
  egress protocol from the same ingest. LVQR's fragment model is the same
  bet expressed differently.

LVQR's Unified Fragment Model plus the CMAF segmenter can beat the Path
model ergonomically if the API is clean. That is a real win, not a wash.

### LiveKit

The SFU runs on top of Pion. Rooms are goroutines. Each Room owns its
participants, their tracks, and the subscription graph. Signaling is a
protobuf-over-WebSocket protocol. Tracks carry simulcast and SVC layers.
The SFU makes forwarding decisions based on subscriber bandwidth and quality
hints. Data channels carry non-media control messages.

LiveKit splits the server into three binaries:

1. **livekit-server**: the SFU, handles ingress and egress of RTP over
   WebRTC.
2. **livekit-ingress**: a separate service that accepts RTMP, WHIP, and
   HTTP URL pull, transcodes the source into a simulcasted WebRTC track,
   and publishes it into a room.
3. **livekit-egress**: a separate service that subscribes to a room or a
   set of tracks, composites them via a Chrome headless render, and emits
   MP4 / HLS / RTMP / S3 / Livestream.

The split is significant: it means the SFU hot path carries only RTP and
never touches fMP4. Ingress and egress amortize CPU-heavy work on separate
machines. LVQR's plan to make the fragment model central is the opposite
bet -- we assume the same binary can serve every protocol at once,
because MoQ subgroups and our data plane do not need per-frame CPU churn
the way WebRTC RTP forwarding does.

LiveKit features LVQR must eventually match for v1.x:

* **Room permissions.** Per-participant subscribe/publish/admin scopes
  with runtime update.
* **Participant SDKs** that abstract reconnection, device switching,
  simulcast layer selection, volume metering, screen share. Our test app
  is a toy by comparison.
* **Agents framework.** A Python or Node process subscribes to a room as
  if it were a participant, runs speech-to-text or an LLM, publishes a
  new track with the response. This is not a content moderation
  afterthought, it is the core of LiveKit's 2025 strategy.
* **Egress-to-storage.** LiveKit's room composite egress records the
  visual mix of all participants plus the mixed audio as a single MP4,
  ready for replay. This solves the "conference recording" problem that
  LVQR's per-track recorder does not address at all.

LVQR's counter-bet: the unified fragment model plus MoQ subgroups can
cover the single-publisher-many-subscriber broadcast case with far less
CPU than an SFU, because we never transcode or re-packetize. LiveKit wins
the many-to-many conference case. Both are legitimate markets. v1.0 LVQR
is a broadcast server, not a conference server.

### OvenMediaEngine

Source tree in three layers: `Providers` (ingest), `Publishers` (egress),
and `Streams` in between. Transcoder sits in the middle. LL-HLS and WebRTC
output are CMAF chunks with matching base media decode times. OVT is
their internal origin-edge protocol, similar in spirit to MoQ but not
standards-based.

OME has been doing LL-HLS and sub-second WebRTC in production longer than
most. The things LVQR should steal from OME's design:

* **Provider/Publisher/Stream separation.** A Stream is protocol-agnostic
  in-memory; every Publisher subscribes to it and emits whatever wire
  format it owns. This is exactly the Unified Fragment Model, seven
  years earlier. The fragment boundary proves out.
* **Signed Policy tokens.** OME accepts signed URLs with expiry and
  optional IP binding. Policy is hashed server-side, no lookup. This is
  cheaper than webhook auth and fits LVQR's SharedAuth trait cleanly.

### SRS

Single-threaded-per-connection using ST (State Threads). RTMP-first design:
every non-RTMP protocol is produced by reading the RTMP stream and
reformatting. HLS, FLV, MP4, WebRTC, SRT all derive from the in-memory RTMP
representation. This is the opposite of the "fragment is the root"
design: SRS's fragment is actually an RTMP tag, and the fMP4 output is a
late-binding conversion.

The SRS approach works well at single-node scale and poorly at clustering.
It is the "good enough for broadcast" server and owns the Chinese market.

The lesson for LVQR: if we cannot make the fragment model cleaner than
"RTMP tag as internal type", we will end up rebuilding SRS in Rust.
The clarifying question: can a Fragment carry enough metadata that every
projection (HLS, DASH, MoQ, WHEP, disk) is obviously derivable from it
with no per-protocol special cases? If yes, we win. If no, we are
reinventing SRS.

### Ant Media Server Enterprise

Java-based, Spring boot, ffmpeg JNI. Cluster mode uses MongoDB for
coordination and a custom gossip layer. Built-in Red5 origin for RTMP
legacy. CMAF LL-HLS. Hardware transcode via VA-API and NVENC through
JNI. Web admin UI with an application-registration model similar to
legacy Red5.

Ant EE is the closest thing to a "turnkey broadcast platform" in the OSS
world, and its feature surface is enormous. LVQR cannot match it by v1.0
and should not try. The goal is to pick the 20 percent of the feature
surface that covers 80 percent of the use cases and ship that cleanly.

### AWS KVS

A fundamentally different architecture: ingest-first, storage-mandatory.
Every fragment is written to the KVS control plane, indexed, and stored.
Real-time playback is a best-effort convenience on top of the durable
store, not the primary mode. The HLS endpoint is generated from the
index at request time.

This is the "archive-first" bet. LVQR's Tier 2.4 archive plus redb index
is architecturally identical. The key difference is that KVS's customers
pay per GB for storage and egress; LVQR's will self-host on disk and
pay zero marginal.

If LVQR's archive + DVR scrub works as designed, we can credibly tell
a customer "take your KVS bill, run LVQR on one beefy box, keep 90
percent of the capability, pay 5 percent of the cost". That is a real
story. It is also the only credible "buy LVQR instead of X" pitch we
have for v1.0.

### Janus and Jitsi

Both WebRTC-only. Janus's plugin architecture is a lesson in extension
points: every feature is a plugin with a stable C ABI. This is the
extensibility we do not yet have in Rust: we have object-safe traits, but
no dynamic loading and no ABI. LVQR's extension story right now is "write
Rust, add a feature flag". That is enough for v1.0. It will not be enough
for a plugin ecosystem by v2.0.

Jitsi's Octo is instructive: Octo is a multi-region routing protocol that
lets Jitsi route a conference's RTP across multiple geographically
distributed bridges. It is a precedent for "cross-node relay over a
custom protocol", which is what LVQR's Tier 3 chitchat cluster plus
MoQ-over-MoQ proposes. The shape is the same; MoQ-over-MoQ should be
easier because MoQ was designed as a pub/sub protocol from day one.

### moq-relay (upstream)

Not a competitor but worth understanding. The reference relay is small,
maintained by one person, and still churning on the wire format. LVQR's
`lvqr-relay` wraps `moq-native`, which in turn uses `moq-lite`. We pay
the cost of upstream churn today and will pay it until the MoQ draft
ships as an RFC. The `lvqr-moq` facade crate (Tier 2.1) is the mitigation
that keeps the churn radius to one file. This is the single most important
hedge in the plan.

## The Five Strategic Bets LVQR Is Making

### Bet 1: MoQ is going to win browser-origin live video

**Confidence**: medium-high. Chrome has shipped WebTransport. Twitch and
Cloudflare are pushing drafts. Luke Curley's moq-lite is a plausible
simplification of moq-transport. Meta and Apple are at the table. The
IETF working group is not stalled.

**Risk**: moq-transport and moq-lite are different specs, and we bet on
moq-lite. The broader industry may pick moq-transport. The mitigation is
the `lvqr-moq` facade: if we need to swap the underlying crate, the
radius is one file.

**What invalidates this bet**: Chrome deprioritizes WebTransport. A
competing protocol (e.g. RTP over QUIC from the IETF side) wins in the
browser. Or the market decides LL-HLS is good enough and sub-second
doesn't matter. The first two are unlikely. The third is the real
risk, and we should have a pure LL-HLS story by M1 anyway.

### Bet 2: Rust memory safety plus perf is worth the ergonomic hit

**Confidence**: high. The buyer side of the market is increasingly
paying for memory-safe infrastructure because of CISA and EU
Cyber Resilience Act pressure. Go is the incumbent safe choice.
Rust is faster, especially on the data plane. Go's SFU latency floor
is dominated by GC and goroutine scheduling; Rust's is dominated by
syscalls.

**Risk**: the buyer market is broader than the security-pressured tier,
and most buyers do not care what language their media server is in.
MediaMTX proves this: Go is fine for most operators. Rust wins only when
either (a) the perf matters at the edge of the envelope, or (b) the
buyer has a Rust platform team.

### Bet 3: a unified fragment model can project cleanly to HLS, DASH, MoQ, WHEP, archive

**Confidence**: medium. OME proved the architecture is viable in C++.
Every server that does many protocols has some internal abstraction.
The LVQR flavor is CMAF fragments with track and group metadata as the
root, with MoQ subgroups and HLS partial segments as projections.

**Risk**: DRM boxes (senc/saiz/saio), variable fragment durations,
multi-audio rendition groups, late-binding subtitle tracks, SCTE-35 ad
markers, and HLS discontinuities all add up to a fragment model that
is not as unified as it looks on paper. We will discover ugly special
cases. The question is whether the special cases live inside the
`lvqr-cmaf` segmenter (fine) or leak into every Publisher crate (not
fine).

**What validates this bet**: when we ship the first HLS + DASH + MoQ
triple-output from one segmenter and the Publisher crates each end up
under 500 lines, we know the model held.

### Bet 4: cross-node MoQ relay-of-relays is the scale-out story

**Confidence**: medium. The shape is right -- MoQ's pub/sub plus
subgroups are designed for relay composition. No one has shipped this
at scale yet. Jitsi Octo proved the general architecture works for RTP;
MoQ should be easier.

**Risk**: chitchat hits a KV scale wall, or MoQ relays churn so fast that
the consistent hash is useless. Tier 3 risk section already flagged
this. Mitigation: build a tiny prototype in week 1 of Tier 3 before
committing to the design.

### Bet 5: WASM per-fragment filters and in-process AI agents matter enough to pay for the sandbox cost

**Confidence**: low-to-medium. The value prop is real (developer loyalty,
provenance, redaction, real-time captions). The execution risk is high
(wasmtime overhead per fragment, GPU access, agent frameworks are a
research black hole). Roadmap Tier 4 MVP caps are the correct
mitigation.

**Risk**: we spend three weeks on WASM filters and the result runs at
100 Mbps per filter when we need 1 Gbps. Tier 4 week 1 benchmark gate
catches this. If we blow the budget, ship filters as control-plane
policies only and defer hot-path filtering to v2.

## What LVQR Has That Matters Today

After Tier 0:

1. **Working RTMP-to-fMP4-to-MoQ-to-WebSocket pipeline** with real
   integration coverage. Every other server on the list does some
   version of this. Ours is Rust and has one unified origin. That is
   a foundation, not a feature.

2. **A pluggable auth trait** (`lvqr_auth::AuthProvider`) with static
   token, noop, and feature-gated JWT providers. The trait shape is
   correct and maps onto Admin, Publish, and Subscribe scopes. LiveKit
   and MediaMTX have equivalent shapes. We are at parity on the shape
   but not the providers.

3. **An event bus** (`lvqr_core::EventBus`) used for cross-subsystem
   lifecycle signaling (broadcast started/stopped, viewer joined/left).
   After Tier 0, the RTMP bridge, the WS ingest session, and the
   recorder all agree on this bus. This is the control plane half of
   the "split control and telemetry" decision from the roadmap.

4. **Peer mesh coordinator** (`lvqr_mesh::MeshCoordinator`) with tree
   assignment, orphan reassignment, and dead-peer detection. No other
   server in this audit has a built-in peer relay concept. It is
   not yet wired into real WebRTC DataChannel media forwarding, but
   the coordination layer is designed and tested.

5. **`lvqr-cli` as a composition root** that actually wires the auth
   provider, event bus, mesh coordinator, admin API, metrics, recording,
   relay, and RTMP ingest into one process. The shutdown race is fixed
   and `tokio::join!` drains all subsystems. This is production-grade
   orchestration; most Tier B servers from the audit do worse.

6. **Real RTMP -> WS end-to-end integration test** with zero mocks, zero
   helper-function-in-isolation assertions. This is the smallest working
   demonstration of the full data path and it runs in under a second.

Everything else the roadmap says LVQR has is either scaffolded or aspirational.

## What LVQR Does Not Have That It Must Ship by v1.0

Ranked by "how loudly the absence embarrasses us in an evaluation":

1. **HLS and LL-HLS egress.** Every serious evaluator will ask "can it
   serve HLS?" and the answer being "no, we only do MoQ and WS" is not
   acceptable. Tier 2.5 is the quagmire item for good reason.

2. **WHIP and WHEP.** These are the browser WebRTC standards, supported
   by OBS and Chrome out of the box. Not having them means our demos
   lose to MediaMTX every time.

3. **DASH egress.** Less critical than HLS but required for any
   non-Apple ecosystem buyer.

4. **SRT ingest.** Broadcast-encoder interop. Without SRT we cannot
   accept feeds from professional broadcast gear. 

5. **RTSP server.** Needed for IP camera feeds and ONVIF integration,
   the entire surveillance and physical security market.

6. **HEVC and Opus codec support.** H.264 plus AAC is 2014-era
   coverage. No AV1 is OK for v1.0. No HEVC is embarrassing by 2026.

7. **Disk archive with redb index and DVR scrub API.** This is the
   "KVS-but-self-hosted" wedge. It is also already in Tier 2.4, but
   the redb index piece must ship at the same time as the recorder
   or the story does not hold together.

8. **Hot config reload.** Every competitor in Tier A supports SIGHUP
   with safe-reload of auth tokens, log levels, and rate limits.

9. **Real observability.** Prometheus metrics alone are not enough.
   Grafana dashboards shipped alongside the binary, tracing with
   OpenTelemetry, cardinality caps, and at least one alert pack are
   table stakes for ops teams.

10. **A JS SDK that is not a toy test app.** `@lvqr/player` and
    `@lvqr/core` exist but the API surface does not cover reconnect,
    device enumeration, adaptive bitrate selection, stats reporting,
    or server-side error mapping. It is a demo, not an SDK.

## What LVQR Must NOT Try To Build for v1.0

This is equally important. The roadmap is honest about this but the audit
should reinforce it.

* **No SIP.** The ecosystem is thin (ezk-sip, none of the alternatives
  production-ready), the market is fragmented (contact centers have
  their own stacks), and the feature cost is enormous. Ship a webhook
  hook so an external SIP gateway can bridge in, and defer.
* **No full conference SFU.** LVQR is a broadcast server, not a
  conference server. Trying to do what LiveKit does with a fraction of
  the engineering budget is how projects die.
* **No SDKs for every platform.** Ship JS and maybe Rust by v1.0.
  iOS/Android/Swift/Kotlin/Flutter wait until v1.1 when there is a
  customer asking for them.
* **No web admin UI beyond the test app.** A separate package in Tier 5.
  The JSON API is the contract.
* **No transcoding in-process.** gstreamer-rs is a dependency that
  eats two weeks of build-time and complicates every platform. Tier 4
  optional.
* **No hardware acceleration beyond whatever tokio-uring gives us on
  Linux.** VAAPI, NVENC, QSV are Tier 4 optional MVPs.

## Implications for Tier 1

Tier 1 as written in the roadmap is "test infrastructure". After this
audit, two adjustments:

1. **Add a cross-implementation conformance harness earlier.** Tier 1.7
   in the roadmap proposes diffing LVQR HLS against MediaMTX output.
   That should graduate to a CI job on the first day Tier 2.5 starts.
   We cannot ship LL-HLS without Apple's `mediastreamvalidator` running
   green on every PR. Bake this into the Tier 1 test infrastructure
   crate now so Tier 2.5 does not have to build it later.

2. **Prioritize the fixture corpus over chaos testing.** The roadmap
   lists `lvqr-chaos` as a Tier 1 deliverable. Chaos testing is
   valuable but does not block Tier 2 work the way the conformance
   corpus does. Re-order: lvqr-conformance first, lvqr-chaos after
   the fragment model is stable.

Everything else in Tier 1 stays.

## What To Study In Source (References For Future Sessions)

* **MediaMTX** path manager (`internal/core/path_manager.go`) and its
  `sourceOnDemand` logic for lazy pull.
* **MediaMTX** HLS muxer (`internal/hls/muxer.go`) for the smallest
  working LL-HLS implementation in any language.
* **LiveKit** ingress service (`pkg/ingress`) for how a separate
  binary cleanly delegates RTMP/WHIP/pull work off the SFU hot path.
* **LiveKit** egress service (`pkg/egress`) for how the room composite
  via headless Chrome is actually plumbed.
* **OvenMediaEngine** CMAF packager (`src/projects/modules/rtp_rtcp/`
  and `src/projects/publishers/llhls/`) for an LL-HLS implementation
  with subsecond latency that is production-tested.
* **SRS** RTMP-first design (`trunk/src/protocol/srs_protocol_rtmp_stack.cpp`)
  as the cautionary example of "every protocol derives from one
  central representation" gone wrong because the central rep is too
  low-level.
* **Pion WebRTC** (`webrtc/v3/peerconnection.go`) for how a clean
  WebRTC stack is organized, since LVQR will eventually need one.
* **moq-rs and moq-lite** (`kixelated/moq-rs` on GitHub) for the
  reference wire format and relay design; this is upstream for our
  Tier 2.1 facade crate.

## Bottom Line

LVQR is a credible prototype that has closed its immediate audit
findings, has a defensible set of strategic bets, and has a roadmap
that maps cleanly onto the gaps identified above. The biggest risk
is execution, not design.

The one architectural call that must be correct is the Unified
Fragment Model in Tier 2.1. Get that right and HLS, DASH, MoQ, WHEP,
and the archive all flow from it. Get it wrong and every Tier 2
crate becomes a rewrite.

Tier 1 is about making Tier 2 possible. The audit does not change
what Tier 1 delivers; it only changes the priority of the conformance
corpus and the comparison harness within Tier 1.

Next action: begin Tier 1 with the `lvqr-conformance` crate, proptest
harnesses for the existing FLV and fMP4 parsers, and an ffprobe
validator helper in `lvqr-test-utils`. This is the lowest-risk,
highest-leverage work we can do right now.
